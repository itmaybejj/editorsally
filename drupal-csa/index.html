Key concepts

Good configurations suppress unhelpful alerts, so content authors can focus on issues they should fix.

Understanding key moments in the testing lifecycle helps put configuration options in context:

  1. Gather elements in specified page regions, e.g.:
    body > *:not('.ed11y-element')
  2. Drop elements that match ignore selectors, e.g.:
    .widgets, .utility-menu
  3. Run tests and sync results to the dashboard.
  4. If split configuration is active and the user is a content editor, filter results:
    • Drop results on elements from outside specified content regions, e.g.:
      main, footer
    • Drop results on elements that match content ignore selectors.
    • Drop results from tests set to "Developer only."
  5. Open the panel and draw alerts if configuration matches an auto-open setting.

Dismiss alerts

Tests can be configured in the library as errors or warnings ("manual checks"). Warnings can be dismissed. There are two types of dismissals:

  • "Ignore" is stored as a user preference: the alert is now hidden for the current user, but only the current user.
  • "Mark OK" hides the alert for all users.

Whether a user is able to dismiss an alert is controlled at the role level in Drupal's user permissions. Most authors should be allowed to dismiss alerts. There is no point in assigning someone a manual check if they cannot mark the item as having been checked.

Specify check areas

These configuration settings are the most important!

Set the first to include any parts of the page content authors can edit. E.g., if they can edit the main content area, .sidebar and .footer-column-two, set those selectors as your content check area.

Use the second to "knock out" subsections. E.g., if main contains various widgets and embeds that only developers can modify, list those: .widget-1, .social-embed;

Developer check area

The key thing to know about the developer check areas is that they are not independent of the content check areas. The developer areas are checked, and then those results are filtered to match the content areas.

Any areas that are excluded for developers will also be excluded for content editors. Do not exclude editable content areas!

Specify pages/users

There is not currently a GUI for turning off tests by content type or route. There are two different selector-based configuration items that can accomplish this using selectors:

The first will hide the checker altogether: no tests will run or sync, and the toolbar will not show.

The second will start the checker minimized, and auto-mark (for the current user) all issues as "Ignored," even if they are errors. This allows the tests to run and update the dashboard, and allows the user to open the checker and use its tools.

Turn off tests

When all else fails, both the content and developer tabs have options to turn off tests altogether.
